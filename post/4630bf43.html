RDD

什么是RDD？RDD是只读的、分区记录的集合。RDD只能基于在稳定物理存储中的数据集和其他已有的RDD上执行确定性操作来创建。

RDD 特性：

  1. 分片（Partition）即数据集的基本组成单位。对于RDD来说，每个分片都会被一个计算任务处理，并决定并行计算的粒度。用户可以在创建RDD时指定RDD的分片个数，如果没有指定，那么就会采用默认值。默认值就是程序所分配到的CPU Core的数目。图描述了分区存储的计算模型，每个分配的存储是由BlockManager实现的。每个分区都会被逻辑映射成BlockManager的一个Block，而这个Block会被一个Task负责计算。

  2. 不变（immutable）

spark 组成

术语表

spark 包含多个概念 driver,executor, task, job 和 stage。

术语 含义
Application 基于Spark的用户程序。由群集上的驱动程序driver programe和执行程序executors组成。
Application Jar 包含用户业务逻辑代码的jar包。 这个Jar可以是Fat Jar也就是包含Hadoop或者spark库，也可以是单纯业务相关以及依赖相关的jar包，这个后面细说如何选择。
driver programe 驱动程序， 用户业务代码的主函数，控制整个数据流，同时负责创建spark context
Cluster Manager 集群管理员，是抽象管理资源分配调度角色，可以管理 standalone, mesos, yarn。
deploy mode 发布模式，主要是区分驱动程序部署位置，在“集群”模式下，框架在集群内部启动驱动程序。在“客户端”模式下，提交者在群集外部启动驱动程序。
worker node 工作节点，集群里执行执行程序的机器，节点包含多个执行者
executor 执行者，运行在woker node负责计算跟存储的进程
task 计算单元，属于stage具体任务，被分配到执行者执行
job 由多个计算单元（task）组成并行计算阶段，由spark action（e.g. save， collect）触发
stage 每个job可以拆分成多个task为一组的stage。这是由dag调度算法来拆分job的stage。
DAGScheduler 根据Job构建基于Stage的DAG，并提交Stage给TaskScheduler
TaskScheduler 将TaskSet提交给worker运行，每个executor运行什么task在此分配
  • executor 的作用可以用于跑不同的作业的task，具体根据TaskSchedule如何分配。这比以前mapreduce，每个任务都需要一个进程来跑来的快，同时节省资源。

application/driver/executor

Driver驱动程序是用户编写的数据处理逻辑，这个逻辑中包含用户创建的SparkContext。SparkContext是用户逻辑与Spark集群主要的交互接口，它会和ClusterManager交互，包括向它申请计算资源等。

Executors是分布式计算程序。他们组成了一个用户程序完整的执行流程spark application。

  1. 用户程序创建SparkContext时，新创建的SparkContext实例会连接到Cluster Manager。Cluster Manager会根据用户提交时设置的CPU和内存等信息为本次提交分配计算资源，启动Executor。
  2. Driver会将用户程序划分为不同的执行阶段，每个执行阶段由一组完全相同的Task组成，这些Task分别作用于待处理数据的不同分区。在阶段划分完成和Task创建后，Driver会向Executor发送Task。
  3. Executor在接收到Task后，会下载Task的运行时依赖，在准备好Task的执行环境后，会开始执行Task，并且将Task的运行状态汇报给Driver。
  4. Driver会根据收到的Task的运行状态来处理不同的状态更新。Task分为两种：一种是ShuffleMap Task，它实现数据的重新洗牌，洗牌的结果保存到Executor所在节点的文件系统中；另外一种是Result Task，它负责生成结果数据。
  5. Driver会不断地调用Task，将Task发送到Executor执行，在所有的Task都正确执行或者超过执行次数的限制仍然没有执行成功时停止。

job/stage/task

应用程序可以理解是一个或者多个job的数据流集合。job之间是通过spark action来划分边界。

job 是由spark action来触发的计算阶段。

根据计算依据是否可以并行化来划分stage。所谓计算依据就是说task计算数据源之间是否可以独立计算。因为task计算数据就是rdd数据集是否有计算依赖。

部署

yarn

在YARN中，每个应用程序实例都有一个ApplicationMaster进程，这是为该应用程序启动的第一个容器。该应用程序负责从ResourceManager请求资源。分配资源后，应用程序指示NodeManagers代表其启动容器。ApplicationMasters消除了对活动客户端的需求：启动应用程序的进程可以终止，并且从群集上运行的YARN管理的进程继续进行协调。

集群部署模式 cluster mode

在群集模式下，Spark驱动程序在群集主机上的ApplicationMaster中运行。YARN容器中的单个进程负责驱动应用程序并向YARN请求资源。启动应用程序的客户端不需要在应用程序的生存期内运行。

群集模式不太适合以交互方式使用Spark。需要用户输入的Spark应用程序，例如 Spark Shell 和 pyspark，要求Spark驱动程序在启动Spark应用程序的客户端进程中运行。

客户端部署模式

在客户端模式下，Spark驱动程序在提交作业的主机上运行。ApplicationMaster仅负责从YARN请求执行者容器。容器启动后，客户端与容器进行通信以安排工作。
